# 节点间关系操作手册（Relate）

> 触发信号：**找不到 / 放不下 / 连不起来**。
> 父级入口：`vault-system-design/SKILL.md`

## 定位

Relate = 操作结构-节点间关系的具体实现（拆分 + 连接）。

- **拆分子流程**：从 0 生成结构、原结构"找不到/放不下"时 MECE 重建
- **连接子流程**：修复孤立页/断链等连接退化

处理**一组**文件/条目或**一批断链/孤立页**（与 distill 处理**单个**文件互补）。

**核心原则**：
- 拆分：**MECE 分桶 + 替换测试 + 人介入在思考末尾**
- 连接：**断链/孤立页定位 + 价值评估 + 修复或归档**

### 与 distill 的边界

|| 维度 | `/vault-distill` | relate |
|---|---|---|
| 输入 | 单个素材 | 一组文件/条目，或一批断链/孤立页 |
| 输出 | 进入已有 entity/topic | 新结构 / 修复后的连接图 |
| 触发信号 | 原文需要萃取 | 找不到 / 放不下 / 连不起来 |

## 拆分子流程：归类（6 步）

> 触发：**找不到 / 放不下**信号。

### Step 0：确定范围、力度与积累门槛

确认范围：哪些文件/条目要归类？

**积累门槛**（结构涌现原则——[[通用系统演化模型#结构涌现原则]]：结构不是设计出来的，是从素材中涌现出来的）：

| 素材状态 | 该做什么 |
|---|---|
| **不足**（<5 篇同类，模式未显现） | 继续积累，不急着建结构；素材留在 Inbox/Source |
| **临界**（5-10 篇，共性开始浮现） | 可以启动本子流程，但预期分桶可能需迭代 |
| **充足**（>10 篇，分类轴清晰可辨） | 正常走本子流程 |

素材不足时强行 MECE → 分类轴多半是猜的 → 后续几乎必然推倒重来。

**操作力度**（由记忆负担严重程度决定）：

| 力度 | 触发条件 | 动作 |
|---|---|---|
| **微调** | 单点摩擦，偶尔放错 | 移动个别文件，**不需要本子流程** |
| **重构** | 某块"放不下/看不懂" | 对这一块做 MECE 拆分 |
| **从 0 生成 / 打碎重建** | 记忆负担持续过大，微调重构无效；或全新领域白手起家 | 全量 MECE 重建 |

**何时"打碎重建"**：微调能恢复可检索性吗？否→升级；多处出现记忆负担？→重建；原分类轴本身就错了？→重建。

### Step 1：盘点与共性识别

**用 `Read` 或 `ls` 获取真实文件列表，禁止凭记忆列。**

每项至少包含：文件名/路径、标题、关键词/主题。

扫描清单，识别聚合信号：

|| 维度 | 例子 |
|---|---|
| 讲者/作者 | 三言两语说英语 x8、rickyshou Agentic x5 |
| 系列 | 日期前缀相同的宏观研报 |
| 主题领域 | 摄影子领域：人像、构图、影调 |
| 抽象层次 | 机构 vs 行业、人物 vs 观点 |
| 生命周期 | 进行中 / 观察中 / 已结束 |
| 价值密度 | 方法论稳定 / 热点时效 / 一次性消费 |

向用户汇报识别到的聚合信号，**给出具体证据**（文件名+关键特征）。

### Step 2：MECE 分桶 + 替换测试

选一个**主分类轴**做一级 MECE 分桶：互斥 + 穷尽 + 桶数 ≤ 7。

每个桶需同时满足：**可独立**（脱离整体仍能理解）、**可复用**（未来可能被引用）、**有密度**（含客观边界或共同主题）。

**常用主分类轴**：目标域（素材归属已基本清楚）/ 讲者/作者 / 抽象层次 / 知识类型（Entity/Narrative/Framework）/ 时效性。

给出分桶草案（含每桶的判别特征 + 文件清单；打碎重建时附旧→新映射表）。

**替换测试**（对每桶随机抽 1-2 个）：

> 把这个文件搬到同级别另一个桶，是否仍成立？✅ 成立→边界模糊，需重新划桶；❌ 不成立→桶边界清晰。

多次替换测试失败 → **返回重选主分类轴**。

### Step 3：人审核（硬约束）

**必须暂停，向用户展示分桶草案 + 映射表，等待审核。**

> 一行糟糕的计划会长出几百行糟糕的内容。精力应集中在审核方案，而非事后 Review 执行结果。

审核通过 → Step 4。不通过 → 返回 Step 2 迭代。

### Step 4：执行重组

1. **建目录 / entity**：按需创建子目录或新页面
2. **物理迁移**：`mv` 移动文件（非复制；打碎重建时记录映射表）
3. **双链调整**：检查并修复因迁移产生的断链
4. **frontmatter 更新**：更新 `state/` 标签、修正 `links`
5. **记录演化原因**：在新建目录的 CLAUDE.md 或 `Z98-Design/演化历史.md` 记录本次重组的触发信号和分类轴

### Step 5：验证（熵增四信号）

| 信号 | 验证方式 |
|---|---|
| 找不到 | 新结构下能否 1 次 grep 定位典型条目？ |
| 放不下 | 抽 3 个新条目试投放，是否都有明确去向？ |
| 看不懂 | 新目录的 CLAUDE.md 是否说清了边界？ |
| 连不起来 | 迁移产生的断链是否全部修复？ |

未达标 → 记录问题，回到 Step 2 或 Step 4 补救。

## 连接子流程：断链 / 孤立页修复（3 步）

> 触发：**连不起来**信号——大规模失效 wikilink、孤立页堆积。
> 伴随重组的双链调整属拆分子流程 Step 4，不在此处理。

### Step 1：盘点

- **断链**：列出所有失效 wikilink（目标已迁移/已删除/拼写不一致/锚点失效）
- **孤立页**：列出所有无入链的页面

### Step 2：修复

**优先级**：高（核心 entity）→ 中（热点 narrative / 频繁引用页）→ 低（边缘/已废弃）。

| 情形 | 动作 |
|---|---|
| 目标已迁移 | 更新链接（必要时改锚点） |
| 目标已删 | 评估删除引用 / 建占位页 / 改为文字提及 |
| 孤立页有价值 | 在相关 entity 中补 wikilink 反向链入 |
| 孤立页无价值 | 迁到 10-Source 归档或删除（需与用户确认） |

**人审核（硬约束）**：修复方案 + 影响清单 → 用户确认 → 执行。批量修改前建议先 git checkpoint。

### Step 3：验证

二次扫描：断链和孤立页数量应显著下降；是否产生新的孤立页；高优先级断链是否全修。

## 硬约束

| 约束 | 含义 |
|---|---|
| MECE 必达 | 桶互斥且穷尽；一级桶数 ≤ 7 |
| 替换测试 | 每桶至少抽检 1 次 |
| 人审核前置 | Step 3 前禁止 `mv`、`rm`、建目录 |
| 映射表硬需 | 打碎重建必须记录旧→新映射 |
| CLAUDE.md 跟进 | 新建目录必须同步创建 CLAUDE.md |
| 演化日志 | 重大重组记录到 `Z98-Design/演化历史.md` |

### 跨层协作

- 发现 schema 不一致 → 父 skill 切到 `reference/specify.md`
- 归类到某 entity 内部太满 → 可再次对本 entity 走本手册，或切 `/vault-distill` 细化
