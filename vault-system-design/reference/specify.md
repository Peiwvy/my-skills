# 定义规范操作手册（Specify）

> 触发信号：**看不懂**。
> 父级入口：`vault-system-design/SKILL.md`

## 定位

Specify = 操作结构-定义规范的具体实现。

通过**人机共建对话**，面向真实素材设计或对齐 schema / 模板 / 命名规范。

**为什么 specify 需要全程人介入**：Schema 是元级决策，一个糟糕的 schema 会让几十/几百个页面长得糟糕，且难事后修（需批量改+改引用）。因此人介入从审核阶段前移到**对话全程**。

### 与其他子流程的边界

|| 子流程 | 管 | 不管 |
|---|---|---|
| `/vault-distill` | 单篇路由到已有结构 | 结构本身 |
| `reference/relate.md` | 建/改拆分结构、修复连接 | 单文件内部字段 |
| 本手册 | **Schema / 模板 / 命名** | 实际路由（是谁） |

## 共建流程（5 阶段）

### Stage 1：需求澄清

**不跳步原则**：这一步没搞清楚之前，禁止进 Stage 2。

询问三个问题：
1. **范围**：哪批素材？哪个目录？一个新类型还是现有 schema？
2. **期望输出**：只设计 schema 文档？还是 schema + 批量应用到素材？
3. **锚点**：有没有参考的 schema_xxx？有没有反例"我不想要这样"？

### Stage 2：素材扫描与归纳

**用 `Read` 读取范围内所有（或代表性）素材**，归纳：常见字段 / 字段的常见值 / 子类差异 / 异常模式。

向用户展示归纳结果，**带具体文件引用**。列出值得讨论的决策点。

**硬约束**：
- **禁止凭空归纳**——每个归纳点必须引用具体文件
- **不跨步提方案**——Stage 2 只做观察，方案在 Stage 3

### Stage 3：方案迭代（多轮）

每轮给出 schema 草案（含字段说明 + 本轮 diff + 1-3 个待决问题），简短不啰嗦。

**迭代规则**：
- 用户反馈 → AI 更新草案 → 再次展示
- **共识前禁止修改任何素材文件**
- 用户说"就这样" / "定稿" → 进 Stage 4

### Stage 4：共识后执行

1. **沉淀 schema 文档**：写入 `Z98-Design/2 结构规范层/页面规范/schema_<name>.md`
2. **批量应用到素材**：对范围内每个文件按 schema 调整，保留原内容只调结构
3. **双链调整**（若 schema 影响了页面位置）
4. **报告完成情况**

**硬约束**：应用前最后一次向用户确认；批量修改前建议 git commit checkpoint。

### Stage 5：验证

- 二次扫描全部应用对象，确认 frontmatter / 结构一致性
- 挑 3 个代表性文件向用户展示
- 检查：一致性（100% 符合 schema？）、可读性（3 秒内识别类型？）、可扩展（新来同类内容能按 schema 写入？）

未达标 → 回 Stage 3 继续迭代。

---

## 硬约束

| 约束 | 含义 |
|---|---|
| **素材驱动** | 每个提议必须引用具体素材，禁止凭空设计 |
| **迭代简短** | 每轮 diff-style，不长篇大论 |
| **用户驱动节奏** | AI 不跨步；每个 stage 完成后等用户确认再进下一 stage |
| **共识前不应用** | Stage 4 前禁止修改任何素材文件 |
| **Schema 沉淀** | 最终 schema 必须写到 `Z98-Design/2 结构规范层/页面规范/schema_xxx.md` |
| **兼容已有** | 新 schema 若与已有 schema 有重叠，必须显式说明差异 |
| **回滚准备** | Stage 4 批量应用前建议 git commit checkpoint |

---

## 核心设计原则

**素材是 schema 的证据**：没有素材就没有 schema 设计的合法性——字段需求必须从真实素材中涌现。

**人机共建不是拖延**：多轮对话是因为 schema 决策难回退，不是因为 AI 偷懒。

**收敛而非发散**：每一轮必须让草案更具体、更可执行。

**Schema 是活的**：沉淀不代表固定不变——新素材涌现新需求时，重新走本手册更新。

**显式优于隐式**：schema 把每个字段的含义、是否必填、取值范围写明，禁止"大家默认这样就行"。
