## 统一整理工具

将讲座逐字稿、文档等整理为结构化 Obsidian 笔记。整理细节见 `rules.md`，本文件只讲流程与模式。

> [!warning] 批量任务铁律
> 1. **单次响应最多处理 2 个文件**——避免输出预算稀释
> 2. ==**首件强制 review（≥3 篇必须执行）**==：第 1 篇写完立即停下，主动告诉用户"已完成第 1 篇 [文件名]，请 review 风格，确认后回复『继续』"——**不允许"先写几篇你看看"**
> 3. **禁止灌水**：信息完整性是硬约束，行数比例只是参考

### 使用方式

```
/summarize [路径]                       # 自动判断模式
/summarize [文件夹] 到一个文件          # 强制课程合并模式
/summarize [文件] 补充到 [目标笔记]     # 增量模式
继续 summarize-[任务名]                 # 恢复未完成任务
```

### 一、标准总结流程（所有模式通用）

> [!important] 模式之间只是编排方式不同，**总结质量标准完全相同**——都按以下 4 步走，都遵循 `rules.md`。

#### Step 1: 读全 + 整理

- **必须先确认 `full_length`**：read_file 默认只显示 200 行，超过必须分段读完。已读行数 = full_length 才能开始写
- 按 `rules.md` 整理：信息完整性优先（每句话默认保留，只删口语填充词、重复、寒暄），逻辑链不跳步，数据全保留

#### Step 2: 验证（强制不可跳过）

对比原文与总结，按 `rules.md` 第三节五项检查执行。发现缺失立即补充。

#### Step 3: 输出

- 命名：`讲者姓名-主题关键词-总结.md`，无法判断讲者时用源文件名
- 格式按 `rules.md` 第二节

#### Step 4: 输出前 self-check

> [!important] 交付前自检（按优先级）
>
> **首要硬指标（必须 100%）**
> - [ ] 信息完整性：每个论点 / 机理 / 数据 / 案例 / 反直觉结论都已保留
> - [ ] 数据保留率 100%：数字、年份、百分比、金额、人名一个不漏
> - [ ] ==Frontmatter 字段/格式与源文件对齐==（特别是 `date created` / `date modified` 用 ISO `YYYY-MM-DDTHH:mm` 格式，不只写日期）
> - [ ] ==每个 `##` 下若只有 1 个 `###` → 必须去掉 `###`==（独苗子节没有划分意义，写完后用 `grep '^##' 文件` 核对每个 `##` 下 `###` 数量）
> - [ ] ==原文图片 100% 保留==：原文所有 `![](url)` 图片链接必须按原位置插入到总结中（核对方式：`grep -c "!\[" 原文` vs `grep -c "!\[" 总结`，前者 ≤ 后者）
> - [ ] ==讲者风格锚点 ≥ 3 处==：反问 / 吐槽 / 口头禅 / 第一人称判断，原话保留在 quote 或正文引号中（自检：删掉这些后，读者还能感受到"是这个人在讲"吗？不能 → 必须保留）
>
> **风格指标（参考）**
> - [ ] 行数 ≈ 原文去语气词后的 0.8-1.5 倍（==禁止==为凑下限灌水或为卡上限删信息）
> - [ ] ==高亮== 限三类：金句 / 反直觉 / 关键数据；每节 1-2 处
> - [ ] 重复并列结构（≥3 项相同子结构）已转表格
> - [ ] Callout 节制：平铺叙述用段落，不每段包裹
> - [ ] 标题按原文逻辑断点划分，无"只有一句话的小节"
> - [ ] abstract = 全文思路 + 结论性话语（连贯叙述 1 段，**不是数字 takeaway 清单**）

### 二、模式判断

按优先级匹配：

| # | 模式 | 触发条件 | 关键动作 |
|---|------|---------|---------|
| 1 | **恢复模式** | 用户说"继续" | 读 `task_plan.md` → 找第一个待处理项 → 继续 |
| 2 | **增量模式** | 用户说"补充到 [目标笔记]" | 新资料逐段判归属，补充到已有笔记 |
| 3 | **课程合并模式** | 文件夹 + 用户说"到一个文件"或检测到索引文件 | 按章节分组，subagent 并行处理后合并 |
| 4 | **批量模式** | 输入是文件夹（无合并需求） | 每个文件独立产出 `*-总结.md` |
| 5 | **拆分模式** | 单文件 ≥ 3 万字（约 15000 汉字） | 按主题断点拆 Part，懒加载逐 Part 处理 |
| 6 | **直接模式** | 单文件 < 3 万字 | 一次读完整理，无需进度文件 |

### 三、进度管理（非直接模式通用）

进度文件位置：`.claude/plans/summarize-[源名称]/`

```
├── task_plan.md      # 主状态：基本信息 + 处理进度表 + 恢复指令
└── progress.md       # 处理日志
```

**task_plan.md 处理进度表**：
```markdown
| 序号 | 文件/Part | 状态 | 输出文件 |
|------|----------|------|---------|
| 1 | xxx.md | ✅ 完成 | xxx-总结.md |
| 2 | yyy.md | 🔄 进行中 | - |
| 3 | zzz.md | ⏳ 待处理 | - |
```

**每完成一项**（严格执行）：
1. 写入输出文件
2. 立即更新 `task_plan.md` 状态（⏳→✅，填输出文件名）
3. 立即追加 `progress.md` 记录
4. 不要攒着批量更新；写入失败时**优先补更新进度文件**再继续

**全部完成**：输出"处理 N 个，跳过 M 个"汇总；保留进度文件；课程合并模式清理 `/tmp/summarize_part*.md`。

### 四、Subagent 派发要点

**何时派发**：批量模式 ≥ 4 个文件 / 课程合并模式按章节分组（每批 ≤ 3 个）。

**关键约束**：
- ⚠️ Subagent 必须用 Bash `cat > /tmp/summarize_partX.md << 'EOF'` 写文件（Write 工具在 subagent 中可能静默失败）
- ⚠️ 文件名含特殊 Unicode（如中文引号 `""`）→ 用 `cat -n /路径/前缀*` 通配符读取
- ⚠️ Subagent 完成后只回复"已完成"，不返回完整内容
- 派发前 prompt 必须嵌入 `rules.md` 完整内容

**合并步骤**（课程合并模式）：
1. `ls -lah /tmp/summarize_part*.md` 验证临时文件存在且非空，缺失则重派
2. Bash `cat` 合并：frontmatter → partA → partB → ... → 总结 callout
3. 写入最终文件后清理 `/tmp` 临时文件

### 五、常见问题处理

| 问题 | 处理 |
|------|------|
| 文件不是逐字稿 / 内容为空 | 跳过，在 `task_plan.md` 标记原因 |
| 已有 `*-总结.md` | 跳过（已处理过） |
| 无法判断讲者姓名 | 用源文件名作为前缀 |
| 上下文不够 | 更新 `task_plan.md`，提示用户开新 session 说"继续 summarize-[任务名]" |
| 增量模式找不到归属 | 创建新章节或询问用户 |
| Subagent 返回内容过多 | 检查 prompt 是否要求"只回复已完成" |
| Subagent 写文件失败 | 改用 Bash `cat >` 写文件 |
| Read 读不了特殊字符文件名 | 用 Bash `cat -n /路径/前缀*` 通配符读取 |
| 合并后临时文件缺失 | `ls -lah` 验证，缺失则重派对应 subagent |
