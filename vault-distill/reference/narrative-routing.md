# Narrative 路由手册

> 本手册是 `vault-distill` 主流程在遇到「跨实体叙事」时加载的子规则。
> 触发条件：Step 2 路由决策中判定为 Narrative（非 Entity）。
> 父级入口：`.claude/skills/vault-distill/SKILL.md`

## 定位

约 **10-15%** 的素材走 Narrative 路径。本手册给出 Narrative 内部的**地理归属**、**生命周期**、**跨 narrative / 多归属**、**历史大事反降级**等分流细则。

**准入门槛判据**（复述自主流程）：

> 问自己"单个 Entity 页面能讲清楚吗"——能 → 走 Entity；不能（跨实体传导链 / 多方博弈）→ 走 Narrative。

---

## 1. 地理归属

| 叙事范围 | 去向 | 示例 |
|---------|------|------|
| 单一实体内 | Entity 时间线/截面 | 某公司业绩变化 |
| 单一国家跨实体 | `F31-Macro/CTY-XX/Narrative/` | 美联储独立性博弈 |
| 跨国多方博弈 | `F31-Macro/REG-XX/Narrative/` | 美伊战争、俄乌冲突 |

---

## 2. 生命周期（状态字段）

| 状态 | frontmatter | 内容形态 |
|---|---|---|
| **进行中** | `状态: 进行中` | 基础骨架 + Active 扩展段（时间线、影响追踪、各方立场、情景矩阵、观察窗口） |
| **观察中** | `状态: 观察中` | 基础骨架 + 精简的 Active 扩展段 |
| **已结束** | `状态: 已结束` | 仅基础骨架；时间线可二次萃取到 Entity，Active 扩展段可归档 |

详细写入格式见 [[schema_narrative]]。

**控制规模**：同期 `进行中` 叙事控制在 1-3 个，不滥建。

---

## 3. 跨多个 Narrative 的材料

```
材料涉及多个 active narrative
  ├── 各 narrative 相互独立 → 找主 narrative 归入，次要打双链
  └── 两个 narrative 本质耦合（去掉任一方叙事不成立）
      → 建独立 narrative 页（REG-XX/Narrative/），各方打双链，`状态: 进行中`
      → 已结束后按国家视角拆散到相关 Entity
```

**判断主 narrative**：哪个 narrative 的结论被这篇材料直接推进或修正。
**判断耦合**：去掉其中一个主体，叙事还成立吗？不成立 → 耦合。

---

## 4. 主从归属 + 锚链接模式（多归属事件的去重写法）

一个事件常有 2+ 合理归属（如「星际之门」既属 AI 泡沫 narrative、又属特朗普 2.0 narrative、还属 OpenAI entity），**绝不复制内容到多处**。规则：

```
主归处：完整萃取（含推断、依据、时间线上下文）
从归处：一行摘要 + 锚链接 → 主归处的 heading
```

**写法示例**：

```markdown
# 主归（AI题材估值与泡沫 narrative）
### 2025-01-22 特朗普揭幕星际之门 AI 基建计划
[完整事实 + 推断 🔗 + 机理分析...]

# 从归（特朗普2.0政治博弈 narrative）
- **2025-01-22 星际之门计划揭幕**（参见 [[AI题材估值与泡沫#2025-01-22 特朗普揭幕星际之门 AI 基建计划]]）：OpenAI/软银/甲骨文 5000 亿 AI 基建；马斯克拆台 [财联社](...)
```

**判断主归**：哪个 narrative/entity 的**结论被这个事件直接推进或修正**，选该处为主归。其他只保留**时间线可见性**（一行概括）+ **可达性**（锚链接）。

**禁止反模式**：

- ❌ 同样的详细内容复制到 2+ 页面（违反"单一真源"）
- ❌ 从归处只打双链但不给时间线条目（导致该 narrative 时间线**不自洽**）
- ❌ 使用 `[[页面]]` 双链但不带 `#heading` 锚点（读者需再次搜索）

## 5. 历史大事反降级规则

> **原则**：信息密度 ≠ 事件重要性。原素材对某事件描述简短（如维基/百科一行），**不代表**该事件可降级为 entity 时间线条目。

**触发建 closed narrative 的三个启发问题**：

```
某事件在素材中描述简短，是否建 closed narrative？
  │
  ├── Q1: 未来有更多素材时，会希望写入独立页面吗？
  │   → 是 → 应建 narrative
  │
  ├── Q2: 用户/检索者是否期望此事件有独立页面？
  │   → 是 → 应建 narrative
  │
  └── Q3: 此事件是否可能被多个 narrative/entity 双链引用？
      → 是 → 应建 narrative
```

**三问任一为「是」→ 建 `状态: 已结束` narrative**（即便初版骨架简洁）。

**典型情形**：

- 历史节点事件（改革开放、中美建交、尼克松访华、布雷顿森林、广场协议等）
- 重大社会运动（占领华尔街、阿拉伯之春、BLM 等）
- 标志性金融事件（2008 次贷危机、1987 黑色星期一等）
- 战争/冲突（已结束者仍建 narrative，进行中者更是）

**反面情形**（可直接时间线化）：

- 常规数据发布（CPI、非农、议息会议决议）
- 公司常规事件（裁员、业绩、高管变动）
- 外交例行接触（会晤、通话）

**closed narrative 的最小骨架**见 [[schema_narrative#已结束 · 初建最小骨架（素材有限场景）]]。

---

## 6. 硬规则

| 规则 | 一句话 |
|---|---|
| **素材未覆盖禁止凭先验补** | 建 closed narrative 时，==素材未覆盖的内容禁止凭历史常识补==——即便是维基百科可查的公开事实。统一放在 `> [!note] 待素材补充` callout 中。违反 → Step 4 校验"先验补全"。 |
| **单一真源** | 多归属事件主归详写、从归一行摘要 + 锚链接；不复制完整内容到 2+ 页面。 |
| **锚点必带** | 从归处的双链必须带 `#heading` 锚点，直达主归事件。 |
| **控制进行中叙事数** | 同期 `进行中` ≤ 3 个；超过需回父 skill 重新权衡（可能走 system-design 合并）。 |

---

## 核心设计原则

**准入门槛是可检验的**：能讲清 → Entity；讲不清 → Narrative。不靠感觉。

**单一真源高于方便**：多归属时宁可让从归简短到看不太懂，也不复制内容。

**信息密度 ≠ 事件重要性**：不因素材描述短就降级历史大事。
